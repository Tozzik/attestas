# Аттестация
Представление о проекте

Проект посвящён практическому изучению системы контроля версий Git и моделированию полного рабочего процесса разработки программного обеспечения в командной среде. Цель работы — получить навыки управления репозиторием, работы с ветками, историей коммитов, слияниями, конфликтами, тегами и релизами, используя только командную строку Git.

1. Подготовка окружения и репозитория

На данном этапе была выполнена настройка пользовательских алиасов Git для ускорения работы и сокращения длины часто используемых команд. Далее был создан локальный репозиторий, в который добавлен исходный код проекта на C#. Для предотвращения попадания во временной контроль версий лишних файлов (например, служебных директорий .vs/, папок bin/ и obj/, логов и временных файлов), был создан и настроен файл .gitignore.
Также в репозиторий добавлен файл README.md с кратким описанием проекта и выполнен первый коммит, фиксирующий базовое состояние проекта. Затем был подключён удалённый репозиторий, что позволило подготовить инфраструктуру для дальнейшей разработки.

2. Организация Git Flow

Для управления разработкой была развернута классическая модель Git Flow. Созданы основные ветки main и develop, а также несколько рабочих веток формата feature/* для реализации отдельных функциональных элементов проекта. Разработка велась изолированно в соответствующих ветках, после чего результаты объединялись в ветку develop с использованием команды git merge --no-ff.
История коммитов велась осмысленно, каждый коммит отражал конкретное изменение, что облегчало анализ и навигацию по проекту.

3. Работа с историей и откатами

В рамках изучения механизмов корректировки истории были отработаны способы отката изменений. Выполнялась отмена ошибочных коммитов с помощью команд git revert, git reset и git restore. Отдельно была изучена возможность восстановления потерянных коммитов через git reflog, что является важным инструментом при восстановлении утерянной истории.
Для временного сохранения незакоммиченных изменений использовалась команда git stash.

4. Работа с конфликтами

Была смоделирована ситуация конфликтного слияния путём внесения противоречивых изменений в разные ветки. В процессе слияния Git обнаружил конфликт, который был решён вручную через редактирование файла. После устранения конфликта изменения были зафиксированы коммитом, что продемонстрировало стандартный процесс разрешения конфликтов в командной разработке.

5. Продвинутая работа с ветками

В рамках освоения продвинутых возможностей Git были выполнены операции переноса изменений между ветками при помощи команд git rebase и git cherry-pick.
Также была изучена работа с визуализацией истории, что облегчает анализ структуры ветвления и слияний в проекте.

6. Теги и релизы

Для фиксации стабильной версии проекта был создан аннотированный тег с описанием. Теги позволяют помечать значимые версии и используются при выпуске релизов. Затем теги были отправлены в удалённый репозиторий, что обеспечивает их доступность всем участникам разработки.

7. Анализ истории и завершение работы

На завершающем этапе ветка develop была слита с основной веткой main, что обозначает достижение стабильной версии проекта. Все изменения, включая теги, были отправлены на удалённый репозиторий с помощью команд git push.
Также был подготовлен файл git_script.sh, содержащий полный набор команд, использованных в ходе работы, а также комментарии к каждой команде и итоговый вывод.
